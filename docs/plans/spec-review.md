## 2. API / DB 세부 명세 누락

### 2-1. API 엔드포인트 Request/Response 스펙 없음

현재 SPEC 문서들은 API 호출 트리거와 URL만 명시하며, 실제 페이로드 형식이 없다.

**누락된 항목:**

#### `POST /api/events` (이벤트 생성)

```
Request Body (필요한 명세):
{
  title: string,        // 이벤트 이름 (1~50자)
  dates: string[],      // 선택된 날짜 배열 (YYYY-MM-DD 형식)
  startTime: string,    // 시작 시간 (HH:00 형식)
  endTime: string,      // 종료 시간 (HH:00 형식)
  timezone: string      // IANA 타임존 문자열 (예: "Asia/Seoul")
}

Response (필요한 명세):
{
  eventId: string,      // 생성된 고유 ID
  url: string           // 공유 URL
}
```

#### `GET /api/events/[eventId]` (이벤트 + 응답 조회)

```
Response (필요한 명세):
{
  id: string,
  title: string,
  dates: string[],
  startTime: string,
  endTime: string,
  timezone: string,
  responses: [
    {
      name: string,
      slots: string[]   // 선택한 슬롯 배열 (형식 미명세)
    }
  ]
}
```

#### `POST /api/responses` (가용 시간 제출)

```
Request Body (필요한 명세):
{
  eventId: string,
  name: string,
  slots: string[]       // 선택한 슬롯 배열 (형식 미명세)
}
```

**특히 슬롯 데이터 형식이 전혀 명세되지 않음** — `"2026-03-20T09:00"` 형식인지, `"2026-03-20_09:00"` 형식인지 불명확.

**수정 방향**: 날짜 데이터 형식은 20260320T090000으로 통일. SPEC 문서에 필요 명세 내용을 추가.

---

### 2-2. DB 스키마 명세 없음

TECH-STACK.md에 Prisma를 사용한다고만 나와 있고, 실제 `schema.prisma` 내용에 대한 스펙이 없다.

**필요한 명세:**

```
Event 테이블:
- id (PK, 생성 방식: UUID? nanoid?)
- title
- dates (저장 방식: JSON 배열? 별도 테이블?)
- startTime
- endTime
- timezone
- createdAt

Response 테이블:
- id (PK)
- eventId (FK → Event.id)
- name
- slots (저장 방식: JSON 배열?)
- createdAt / updatedAt

관계: Event 1:N Response
```

**수정 방향**: SPEC 문서에 필요 명세 내용 추가

---

### 2-3. eventId 생성 방식 미명세

PRD/SPEC 어디에도 고유 eventId를 어떻게 생성할지 명시가 없다.

- UUID v4 (`crypto.randomUUID()`)
- nanoid (짧고 URL-safe)
- CUID2

URL 공유가 핵심 기능인 만큼, 짧고 가독성 좋은 방식 선택이 UX에 영향을 미침. 명세 필요.

**수정 방향**: eventId는 nanoid 방식으로 생성하도록 수정

## 3. 엣지케이스 / 동작 명세 부족

### 3-1. 저장 성공 후 화면 전환 동작 미명세 (SPEC-002)

현재 SPEC-002의 "5.1 저장 버튼 클릭"에서 성공 시 "저장되었습니다 ✓" 토스트만 명시되고, 이후 동작이 없다.

**질문:** 저장 성공 후 자동으로 "전체 현황" 탭으로 이동하는가, 아니면 현재 화면을 유지하는가?
**권고**: 저장 성공 후 "전체 현황" 탭으로 자동 전환하는 것이 UX상 자연스러움. 명세 추가 필요.

**수정 방향**: 저장 성공 후 현재 화면을 그대로 유지함.

---

### 3-2. 중복 이름 처리 시점 미명세 (SPEC-002)

PRD에서는 "기존 응답 불러와 수정 모드로 진입"이라고 기술하나, SPEC-002에서는 "저장 시 기존 응답 데이터 덮어쓰기"라고만 되어 있다.

**불명확한 점:**
- 이름 입력 후 즉시 기존 응답을 불러오는가? (입력 시점 조회)
- 저장 버튼 클릭 시 덮어쓰기만 하는가? (조회 없음)
- 기존 응답을 불러온다면, 그리드에 기존 선택값이 미리 표시되는가?

**권고**: 현재 SPEC-002의 "초기값: 빈 값 (이전 세션 로컬스토리지 저장 미지원)" 명세와 "기존 응답 불러와 수정 모드" PRD 내용이 충돌함. 동작 방식을 하나로 확정하고 명세 필요.

**수정 방향**: 기존 응답 불러오지 않음. 기존 응답자와 이름이 같더라도 별개의 응답으로 처리.

---

### 3-3. 타임존 처리 세부 명세 부족

타임존 관련 여러 항목이 미명세:

| 항목 | 현재 상태 |
|------|----------|
| 서버 저장 방식 | UTC 저장? 입력된 타임존 그대로 저장? |
| 슬롯 비교 방식 | 다른 타임존 참여자 간 슬롯 비교 방법 불명 |
| 타임존 드롭다운 목록 | 전체 IANA 타임존? 주요 국가만? |
| 타임존 기본값 | "브라우저 자동 감지"라고만 명시 |

**권고**: MVP에서는 복수 타임존 지원의 복잡도를 피하고, 이벤트 생성 시 설정된 타임존 기준으로 고정 표시하는 방향을 명확히 명세할 것.

**수정 방향**: 서버에는 UTC 기준으로 저장. scr-002의 내 가용시간 응답과 scr-003의 전체 현황에도 타임존 선택 기능을 제공. 즉, UI에는 사용자가 선택한 타임존 기준으로 데이터 보여줌 & 서버에는 모든 입력/응답값을 UTC 기준으로 기록. 타임존 드롭다운 목록은 전체 IANA 타임존을 노출. "도시명 + (GMT+-N)" 방식으로 표기. 타임존 기본값은 사용자가 진입한 브라우저의 타임존을 기준으로 세팅.

---

### 3-4. 초기 로딩 상태 스펙 없음 (SPEC-002)

SCR-002 진입 시 `GET /api/events/[eventId]` 데이터 로드 중 사용자에게 표시되는 UI가 명세되지 않음.

**권고**: 로딩 스피너 또는 스켈레톤 UI 표시 여부 및 방식 명세 필요.

**수정 방향**: 로딩 스피너 노출

---

### 3-5. 이벤트 만료/삭제 정책 미정의

이벤트가 생성 후 영구 보존되는지, 만료 기간이 있는지 명세가 없다.

**권고**: MVP에서는 "무기한 보존"이라고 명시하거나, 90일 등 만료 정책을 결정하여 명세 추가.

**수정 방향**: 생성일로부터 90일까지 보관하고, 90일 이후 이벤트 링크와 해당 이벤트의 응답값 모두 완전 삭제(hard delete). 

---

### 3-6. 선택 슬롯 0개 저장 허용 여부 미명세 (SPEC-002)

이름만 입력하고 슬롯을 하나도 선택하지 않은 채 저장 가능한가?

- 가능하다면: "가능한 시간이 없음"으로 응답이 반영됨
- 불가능하다면: 추가 유효성 검사 필요

**권고**: 동작 방향 결정 후 SPEC-002 Error Handling 섹션에 추가.

**수정 방향**: 저장 불가. 가능한 시간을 선택해달라는 오류 토스트 노출.

---

### 3-7. 모바일 하단 시트 세부 동작 미명세 (SPEC-003)

SPEC-003에 모바일 하단 슬라이드업 시트가 명시되어 있으나 다음이 미명세:

- 슬라이드업 애니메이션 방향/속도
- 시트 외부 탭으로 닫기 시 배경 딤(dim) 처리 여부
- 시트 높이 (`높이 auto, 최대 50vh`는 명시됨)

**수정 방향**: 배경 dim 처리하지 않고, 모바일 슬라이드업 시트로 제공되는 경우 시트에 그림자 효과 추가. 슬라이드업 애니메이션 방향/속도는 적절한 값으로 채워넣어줘.
